# 一秒翻盘 / flip-second — MVP 执行计划

## 1) MVP Scope
- **目标**：今天内做出一个可在手机浏览器/微信内打开、能完整游玩的 H5 原型。
- **核心玩法**：玩家在极短时间内做“翻盘决策”——在倒计时结束前点击正确的“逆转按钮/卡牌”，把失败局面扭转为胜利。
- **本次只做 1 个模式**：`单人闯关 + 分数冲刺`。
- **本次不做**：联网对战、账号系统、排行榜后端、复杂剧情、支付、广告、关卡编辑器。
- **MVP 核心功能**：
  - 开始页 → 教学 → 游戏主循环 → 结算页 → 再来一局
  - 3 个难度阶段（通过节奏加快体现）
  - 10~20 局连续挑战
  - 倒计时、得分、连击、失误次数
  - 失败反馈与轻量翻盘动画
  - 本地保存最高分
- **技术边界**：
  - 优先原生前端或轻量框架实现（如 Vite + Vue/React 均可）
  - 单页应用即可，不要求路由复杂化
  - 资源全部本地静态文件，可直接部署到静态托管

## 2) Gameplay Loop
1. 玩家进入开始页，看到一句话规则：**“在 1 秒内选对翻盘操作，连续翻得越多分越高。”**
2. 点击“开始挑战”后进入 3~5 秒超短教学。
3. 每回合出现一个“危机局面卡”+ 2~4 个可选操作。
4. 回合倒计时开始（早期 1000ms，后期压到 600~700ms）。
5. 玩家在时间内点击正确选项：
   - 成功：进入“翻盘”反馈，得分增加，连击+1，下一回合立即开始。
   - 失败/超时：扣失误次数，打断连击，进入下一回合或直接结算。
6. 持续完成固定回合数，或失误耗尽后进入结算。
7. 结算展示：本局分数、最高分、翻盘成功次数、最佳连击。
8. 玩家可“再来一局”或“返回首页”。

### 推荐数值（今天可直接落地）
- 初始生命/容错：3 次失误
- 总回合数：15 回合
- 初始每回合时限：1000ms
- 第 6 回合后：900ms
- 第 11 回合后：750ms
- 正确得分：100 分
- 连击奖励：每层 +20 分，上限 +100
- 超时按错误处理

### 危机场景题库方向（先做 12~20 条）
统一采用“失败局面 → 正确翻盘动作”结构，减少设计成本。
示例：
- **球要进了** → 门将扑右下
- **电量 1%** → 开省电模式
- **锅要糊了** → 立刻关火
- **老板走来** → 切回汇报页
- **炸弹倒计时** → 剪蓝线
- **考试只剩 1 分钟** → 先写会的题

> 规则要“看一眼就懂”，不要依赖长文本阅读；每题最好 1 张图 + 2~4 个图标按钮。

## 3) Scene / Page List

### A. 启动 / 首页
**目的**：快速进入游戏。
- Logo / 游戏名
- 一句话卖点
- `开始挑战` 主按钮
- `玩法说明` 次按钮
- 角落显示“最高分”

### B. 玩法说明弹层 / 短教学页
**目的**：10 秒内教会玩家。
- 说明：在倒计时结束前点击正确翻盘动作
- 演示 1 个样例题
- `我懂了，开始` 按钮

### C. 游戏主场景
**这是 MVP 核心页面，必须优先完成。**
- 顶部 HUD：当前分数 / 连击 / 剩余失误
- 中部：危机局面卡（标题 + 插图/emoji + 极简描述）
- 倒计时进度条/圆环
- 底部：2~4 个操作按钮
- 选中后立刻反馈：正确/错误、轻微震动感/颜色闪动
- 自动切到下一题，不弹复杂中间页

### D. 回合反馈层（可与主场景合并）
**目的**：增强爽感，但不能拖慢节奏。**
- 正确：显示“翻盘成功！”
- 错误：显示“慢了/选错了”
- 200~400ms 内自动消失

### E. 结算页
- 本局总分
- 最高分
- 成功翻盘次数
- 最佳连击
- 文案评级（如：`差点意思 / 有点东西 / 真能翻`）
- `再来一局`
- `返回首页`
- 可预留 `分享战绩` 按钮占位（今天不接真实分享）

## 4) Acceptance Criteria for Today

### Playable 基线
- [ ] 手机竖屏下可正常打开和游玩
- [ ] 从首页能进入游戏并完成一整局
- [ ] 至少有 12 个可轮换题目
- [ ] 每题有明确唯一正确答案
- [ ] 存在倒计时压力，超时会判失败
- [ ] 有分数系统、连击系统、失误上限
- [ ] 游戏能结束并进入结算页
- [ ] 支持“再来一局”完整重开
- [ ] 最高分保存在 localStorage

### 体验基线
- [ ] 单局时长控制在 30~60 秒
- [ ] 单手点击区域足够大，按钮不误触
- [ ] 文字在 390px 宽度设备上无需横向滚动
- [ ] 页面切换和反馈不明显卡顿
- [ ] 规则无需额外解释，首次玩家 1 局内能理解

### 开发完成定义
- [ ] 代码结构至少分为：数据题库、状态管理、场景 UI、结果计算
- [ ] 题库配置可通过 JSON/TS 数组快速扩充
- [ ] 所有核心流程无阻塞 bug（开始不了、卡死、无法结算等）
- [ ] 可部署为静态网站并直接分享链接测试

## 5) Asset / Style Guidance

### 视觉方向
- **关键词**：微信 H5、轻量、夸张、短平快、强对比、爽感反馈。
- **整体气质**：像“魔性小游戏 + 短视频梗图”的混合，不做重度写实。
- **推荐风格**：
  - 大标题
  - 高饱和撞色
  - 卡片式布局
  - 粗圆按钮
  - 强烈倒计时警示色

### UI 建议
- 主色：亮黄 / 荧光绿 / 热血红 三选二
- 背景：深色底配高亮卡片，更能突出倒计时和按钮
- 字体观感：粗体优先，标题字大，正文尽量不超过两行
- 按钮高度建议 >= 48px，点击热区尽量 >= 56px
- 动效控制在 150~300ms，追求“脆”而不是“飘”

### 资源策略
- 今天先用 **纯色块 + emoji + 简单图标** 完成，不等正式美术。
- 插图优先使用：
  - emoji
  - SVG 图标
  - 几何形状
- 音效可选：
  - 正确：短促“啪/叮”
  - 错误：短促“嘟”
  - 倒计时结束：轻警报
- 若音效来不及，优先做视觉反馈，不要卡在素材制作上。

### 文案原则
- 每条题面 4~10 个字优先
- 操作按钮 2~5 个字优先
- 结果反馈要短：`翻了！` / `晚了！` / `错了！`
- 避免长段说明，节奏比世界观更重要

### 题库设计原则
- 看图即懂 > 阅读理解
- 正确答案要合理，但错误选项要“看起来也像那么回事”
- 题目类型前期控制在 3 类内：
  - 反应型（立刻点对）
  - 常识型（选正确补救动作）
  - 节奏型（后期加快时限）

## 今天的落地顺序（建议开发顺序）
1. 先完成 `首页 + 主游戏页 + 结算页` 骨架
2. 接入题库数据和回合切换
3. 接入计时器、判定、得分、失误
4. 做正确/错误反馈
5. 接入 localStorage 最高分
6. 最后再补教学页、动效和音效

> 判断标准只有一个：**今晚必须能把链接丢进微信里，让人点开就能玩一整局。**
